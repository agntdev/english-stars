# fix-4d45d7c4294c73f7 — Reminder storage committed before scheduling succeeds — partial failure leaves inconsistent state

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 ENSTAR

In the text-input handler (`src/bot.ts:412-429`):
1. `await reminderStorage.write(userId, { time, cadence })` — storage committed.
2. `await scheduleReminder(userId, time, cadence)` — calls `upsertJobScheduler`.
3. `await ctx.reply(...)` — success message.

`scheduleReminder` (`src/reminder.ts:33-51`) try-catches `removeJobScheduler` but leaves `upsertJobScheduler` **unguarded**. If `upsertJobScheduler` fails (Redis error, connection loss), the error propagates: storage already has the record (persistent) but no scheduler exists (job lost). The user never sees the success reply; `bot.catch` sends a generic error. On next `/reminders`, the stale-but-saved time is displayed as if the reminder is active when it isn't.

The fix: either schedule-then-write (roll back on failure), or wrap both in a transaction with rollback on error.
